1. What does this feature do?

When cache.api.etag.enabled=on (default for single-node), metadata and composite API GETs get a content-version ETag. Browsers and clients send If-None-Match; if nothing relevant changed, the server returns 304 Not Modified before running the heavy controller path.

Invalidation is driven by a JDBC DML observer: INSERT/UPDATE/DELETE on observed tables bumps small in-memory version counters. The next ETag for that type changes, so clients re-fetch once.

  • One config switch gates both the observer and the ETag interceptor.
  • CI runs a dedicated cache-api-test e2e job (JUnit group cache).
  • Metrics (optional): monitoring.cache.etag.enabled=on.

2. What it does not do

Important for the RAM discussion This is not a response-body cache (not Redis of JSON payloads, not a CDN blob store).
  • Does not store response bodies.
  • Does not cache tracker/analytics bulk data paths the same way.
  • Does not share version counters across cluster nodes (see clustering below).

4. Will it use a lot of RAM?

Short answer for the team No. There is no response-body cache. Under the same load, the process allocated less with the feature ON (more short-circuit 304s), not more.

What lives in memory

Structure What it is How it grows
entityTypeVersions Map FQCN → AtomicLong One entry per entity type ever bumped
namedVersions Map name → AtomicLong Tiny fixed set (e.g. apps, staticContent)
allCacheVersion One AtomicLong Manual maintenance clear only
pendingBatches In-flight DML per JDBC connection Deduped by table+op; swept after 5 min

Design bound: on the order of kilobytes, not “size of all API JSON”. Unit test: 10 000 bumps of the same type → map size stays 1 (LocalETagServiceCardinalityTest).

Observability gauges (optional)

When monitoring.cache.etag.enabled=on, Micrometer exposes:

  • dhis2_etag_entity_versions_size
  • dhis2_etag_named_versions_size
  • dhis2_dml_observer_pending_batches
  • dhis2_dml_observer_pending_events_total

These should stay small and flat in production. Full write-up: MEMORY-etag.md.

Honesty box We have not yet published a multi-hour soak chart. The bound is by design + unit tests + alloc A/B under load. A long soak would refine confidence, not rewrite the model.

5. Clustering decision

Version counters are process-local. In a multi-node DHIS2 cluster they would not stay coherent without a shared version store (not built yet).

Product decision (2026-07-10) Keep default on for single-node (the common case). If DHIS2 clustering is configured (cluster.members + cluster.hostname), the ETag cache and DML observer are forced off at startup with a WARN log (ApiETagCacheActivation. Shared-version propagation can come later.

6. How to verify yourself

Re-run latency A/B (Docker)

cd dhis-2/dhis-test-performance
# Build a local core image of this branch first, then:
DHIS2_IMAGE=dhis2/core-pr:etag-ab-local \
PROFILE=load WARMUP=1 MEASURED=2 FAST=false CAPTURE_SQL=1 \
./scripts/etag-ab-benchmark.sh

Re-run alloc / CPU flamegraphs

# Allocation (memory story)
PROF_ARGS="-e alloc" CAPTURE_SQL=1 \
DHIS2_IMAGE=dhis2/core-pr:etag-ab-local \
PROFILE=load WARMUP=1 MEASURED=1 FAST=false \
./scripts/etag-ab-benchmark.sh

# CPU
PROF_ARGS="-e cpu" ... same as above

Unit tests (cardinality / concurrency)

mvn -pl dhis-support/dhis-support-cache-invalidation test \
  -Dtest=LocalETagServiceCardinalityTest,LocalETagServiceConcurrencyTest

Cache e2e suite

# Against a live instance with cache.api.etag.enabled=on
cd dhis-2/dhis-test-e2e
mvn -Pcache test -Dinstance.url=http://HOST:PORT/api ...

CI: parallel job cache-api-test in run-api-tests.yml.
