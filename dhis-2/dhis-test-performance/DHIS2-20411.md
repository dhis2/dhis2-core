# DHIS2-20411

Analyzing tracker request logs from nginx during performance tests.

## Run Locally

### Performance Tests with Logging

To run performance tests with nginx/ELK logging enabled:

```sh
# Start services with logging profile
docker compose --profile logging up -d

# Run performance tests against nginx (port 8081)
DHIS2_URL=http://localhost:8081 ./run-simulation.sh
```

* DHIS2 is accessible directly on port 8080 (no logging) and via nginx on port 8081 (with logging)
* login to Kibana at http://localhost:5601/app using user `elastic` and password `changeme`
* discover logs for index `tracker-local`

### Pre-built Kibana Dashboards

The logging profile automatically imports two dashboards for Gatling test configuration:

1. **Gatling Think Time Analysis** - Extract pause() values for realistic user behavior
   * Shows response time metrics and session request sequences
   * Export session data to calculate think times: `gap_between_requests - response_time`
   * Use median for `.pause(median)` or min/p95 for `.pause(min, p95)` in Gatling

2. **Gatling Injection Profile (Little's Law)** - Extract concurrent users and arrival rates
   * Shows concurrent sessions over time (L in Little's Law)
   * Calculates metrics needed for `rampUsersPerSec()` and `constantUsersPerSec()`
   * Follows Little's Law: `L = λ × W` where:
     * L = Concurrent users (from session counts)
     * λ = Arrival rate (L / W)
     * W = Response time + Think time

3. **Gatling User Arrival Rate** - Extract user arrival rates from nginx logs
   * Shows new users per second based on session IDs
   * Validates arrival rate calculation using Gatling tests
   * See [ANALYSIS-ARRIVAL-RATE.md](ANALYSIS-ARRIVAL-RATE.md) for detailed guide

**Access dashboards**:
* Navigate to **Analytics → Dashboard** in Kibana
* Or use Discover with the **Tracker Requests** data view

**Customization**: Dashboards are imported from `docker/kibana-dashboards/*.ndjson` files.
Modifications in Kibana persist in the `kibana` volume. After editing dashboard files, reimport
them without restarting:

```sh
./kibana-reimport-dashboards.sh
```

To version control changes made in Kibana UI, export updated dashboards and commit the NDJSON
files.

### Performance Tests without Logging

The default docker compose setup runs without the nginx/ELK stack for maximum performance:

```sh
docker compose up
```

This exposes DHIS2 directly on port 8080 without nginx proxy.

### Analyze Only

Starting the ELK stack is enough if you only want to analyze data you have already indexed:

```sh
docker compose --profile logging up es kibana
```

### Clean data

If you changed the nginx log format and want to start fresh you can stop the nginx and vector
containers and delete the volume which stores the nginx logs (or delete the file in it).

```sh
docker compose rm -sf nginx vector
docker volume rm dhis-test-performance_nginx
```

Then delete the elasticsearch index:

http://localhost:5601/app/management/data/index_management/indices

and start nginx and vector again.

### Troubleshooting

Elasticsearch might fail to start due to this error:

https://discuss.elastic.co/t/eck-elasticsearch-pod-not-starting-on-aks-max-virtual-memory-areas-vm-max-map-count-65530-is-too-low-increase-to-at-least-262144/305788

Apply the suggested fix:

```sh
sysctl -w vm.max_map_count=262144
```

## Nginx Config

Test the nginx config:

```sh
docker compose exec nginx nginx -t
```

Reload the nginx config:

```sh
docker compose exec nginx nginx -s reload
```

Stream tracker request logs:

```sh
docker compose exec nginx tail -f /var/log/nginx/tracker_access.log
```

Copy tracker request logs:

```sh
docker compose cp nginx:/var/log/nginx/tracker_access.log ~/tracker_access.log
```

## Vector Config

Restart vector if you change its config:

```sh
docker compose restart vector
```

Vector does have `--watch-config` but I prefer the explicit reload for this use case.

### Access log pattern

nginx does not support logging as JSON well. There are some workarounds but in order to not put any
extra load or odd logic into implementations nginx configurations we opted to log using a custom
csv parsable pattern. The separator `|` is used as its unlikely part of any logged value.
